PhysioCheck ML and deep learning model reference

What ML/DL does in PhysioCheck

PhysioCheck uses ML at three distinct layers — pose estimation (pre-trained, no training needed), feature-based rule logic (no ML), and two trainable deep learning models that grow in sophistication across phases.
Layer 1 — Pre-trained
MediaPipe Pose
Google's pre-trained landmark detector. Zero training required. Outputs 33 landmarks per frame.
Layer 2 — Rule-based
Angle & threshold engine
Pure math. Computes joint angles, velocity, ROM. No ML — deterministic comparisons against therapist-set thresholds.
Layer 3 — Phase 1 model
GRU classifier
Trained on labelled exercise sequences. Classifies posture state + error type in real time.
Layer 4 — Phase 2 model
Transformer + TCN
Advanced temporal model for long-sequence understanding, personalized correction, and injury risk prediction.

Phase 1 — GRU sequence classifier

Why GRU?

Gated Recurrent Unit

Exercises are time-series data — a squat isn't a single frame, it's a sequence of frames. GRU processes this sequence step-by-step and remembers relevant context (e.g. "knee was bending") without the overhead of LSTM. It's faster and has fewer parameters, which matters for <100ms real-time inference.

Lower latency than LSTM Fewer parameters Handles variable-length sequences Proven on gesture/motion tasks

GRU architecture

Input shapeBatch × T (30 frames) × F (18 features)
GRU layer 1128 hidden units, return sequences
GRU layer 264 hidden units, return last hidden state
Dropout0.3 between GRU layers
Dense layer64 units, ReLU
Output layer5 units, Softmax
LossCategorical cross-entropy
OptimizerAdam (lr=1e-3)
Output classesCorrect · Knee valgus · Rounded back · Incomplete ROM · Risky movement

Phase 2 — Transformer + TCN

Why Transformer?

Multi-head attention

GRU is great for short sequences but struggles to "look back" far in time — e.g. noticing that someone's hip dropped 15 frames ago causes a knee error now. Transformers with self-attention handle this by relating every frame to every other frame in the sequence. The result is more accurate phase detection and better capture of compound movement errors.

Long-range temporal dependencies Parallelisable training Better context per frame Scales with data

Why TCN alongside Transformer?

Temporal Convolution Network

TCNs are 1D dilated causal convolutions — they are exceptionally fast at inference (no recurrence, fully parallelisable), and they capture local temporal patterns (like the velocity spike just before an error) that Transformers sometimes miss. The two models are complementary: Transformer for global context, TCN for local motion patterns.

Dilated causal convolutions No recurrence — fast inference Local motion pattern detection Works well on small datasets

Transformer + TCN architecture

InputBatch × T (90 frames) × F (18 features)
Positional encodingSinusoidal, added to input
Transformer encoder4 layers, 4 heads, d_model=128
TCN branch4 dilated conv layers (kernel=3, dilations 1,2,4,8)
FusionConcatenate Transformer + TCN outputs → Dense 128
Phase head5-class softmax (exercise phases)
Error headMulti-label sigmoid (error types)
Risk headScalar sigmoid (injury probability 0–1)
LossWeighted sum: CE + BCE + MSE

Optional Phase 3 — Graph Neural Network

Phase 3

The human body is a graph — joints are nodes, bones are edges. A GNN (specifically ST-GCN — Spatial-Temporal Graph Convolutional Network) can model joint interdependencies directly. For example, it naturally captures that "knee valgus is caused by weak hip abductors" because those joints are connected in the graph. This is research-level and best approached after Phase 2 data is collected.

ST-GCN architecture Models joint-bone relationships Best for injury prediction Needs 3D pose input

End-to-end training pipeline

Step 1
Data collection
Record subjects doing exercises via webcam. Collect from public datasets. Annotate error types per frame.
Step 2
Pose extraction
Run MediaPipe on all videos. Extract 33 landmarks per frame. Store as numpy arrays with timestamps.
Step 3
Feature engineering
Compute angles, velocity, ROM, symmetry. Normalise. Build sliding window sequences of T frames.
Step 4
Labelling
Physio annotates each sequence window: exercise phase + error type + severity. Tool: Label Studio or custom UI.
Step 5
Augmentation
Mirror sequences, add Gaussian noise to landmarks, temporal scaling, shift sequences. Doubles dataset size.
Step 6
Train / val / test split
70/15/15 split. Split by subject (not by frame) to prevent data leakage across people.
Step 7
Training
PyTorch training loop, early stopping on val loss, cosine LR schedule, mixed precision (fp16).
Step 8
Evaluation
Accuracy, F1 per class, confusion matrix. Test on unseen subjects. Check latency on target hardware.
Step 9
ONNX export
Export trained model to ONNX. Run ONNX Runtime for fast CPU inference in the FastAPI backend.

Dataset requirements & sources

The most important thing: split your dataset by subject, not by frame. If frames from the same person appear in both train and test sets, your model will look much better than it really is — it has memorised that person's movement style.

Public datasets alone won't be enough — none have error annotations at the granularity PhysioCheck needs (e.g. "knee valgus at frame 43 of this squat"). You must collect and annotate your own data.

Class imbalance handling

Correct posture reps will outnumber specific error types. Handle this with:

Weighted cross-entropy loss SMOTE for minority classes Oversample error sequences Focal loss (Phase 2)

Real-time inference architecture

Inference flow per frame

1. Webcam frameCaptured at 30 FPS in browser
2. MediaPipe (browser)Run MediaPipe Pose JS — landmarks extracted client-side
3. Feature compute (browser)Angles + velocity computed in JS — lightweight math only
4. WebSocket sendSend feature vector (not raw video) to backend
5. Sliding window bufferBackend maintains last T frames per session
6. ONNX inferenceRun GRU/Transformer on T×F buffer — <30ms
7. Prediction + feedbackClass label + confidence + feedback text returned via WebSocket
8. RenderClient renders skeleton overlay + text/voice feedback
Key design choice: run MediaPipe in the browser (JS), not on the server. This avoids streaming raw video (expensive bandwidth). Only the feature vector (~18 floats per frame) travels over WebSocket.

Calibration step (per user, first session)

Before scoring, the system captures 5 "reference reps" of each exercise from the user. This sets a personal baseline — what "correct" looks like for their body. Angle thresholds are adjusted ±10% from this baseline. This handles body structure variation without retraining the model.

Personal ROM baseline Threshold auto-adjustment No retraining needed Stored per user in DB

ClothingLoose clothing reduces landmark quality — test for degradation

Real-world validation (before clinical deployment)

Before any clinical claim is made, PhysioCheck needs observational validation with physiotherapists confirming the system's feedback matches their clinical judgement across ≥ 50 patients and ≥ 500 sessions. This is a prerequisite for any medical or healthcare use-case positioning.

Physio agreement rate ≥ 80% No missed high-severity errors IRB / ethics approval Longitudinal ROM tracking validated
Do not ship injury risk prediction as a clinical safety feature without clinical validation. In the product, frame it as "movement quality guidance" rather than medical advice until validated.
